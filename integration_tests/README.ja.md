# 統合テスト (Integration Tests)

このディレクトリには、リポジトリのAIエージェント設定を安全に検証するために設計された、コンテナベースの統合テストスイートが含まれています。

## 概要 (Overview)

統合テストでは、以下の事項を確認します：

1.  **セキュリティフック**（`gh-safeguard.sh`など）がデータ搾取コマンドを正しくブロックすること。
2.  **エージェント設定**が有効であり、これらのフックを正しく登録していること。
3.  **エージェントCLI**（Claude、Gemini、Cursor、Codex）が、実際の（ただし隔離された）環境でこれらの設定を尊重すること。

## テスト戦略 (Testing Strategy)

テストランナー（`verify.sh`）は、3つの異なるフェーズで動作します：

### Phase 1: フックの直接検証 (Direct Hook Verification)

このフェーズでは、`gh-safeguard.sh`スクリプトを直接ユニットテストします。ダミーのJSONペイロード（さまざまなエージェントのスキーマをシミュレート）をフックに渡し、ブロックされたコマンドに対しては終了コード2を、安全なコマンドに対しては終了コード0を返すことをアサートします。

### Phase 2: 設定の検証 (Configuration Verification)

このフェーズでは、`jq`と`grep`を使用して、コンテナ内のシンボリックリンクされた設定ファイルを検査します。各エージェントが、このリポジトリで提供されるセキュリティフックまたはサンドボックス機能を使用するように明示的に設定されていることを検証します。

### Phase 3: エージェントCLIの調査 (Agent CLI Probes)

このフェーズでは、実際のエージェントCLIコマンド（`claude -p`、`gemini prompt`など）の実行を試みます。

- **セキュリティブロック**: フックがコマンドを正しく傍受した場合、エージェントは終了コード2を返します（成功）。
- **認証エラー**: テストはダミーのAPIキーで実行されるため、終了コード1を返す場合があります。これは「ソフトパス（不完全な実行）」として扱います。なぜなら、エージェントCLIがインストールされており、認証で失敗する前に設定されたフックを実行しようとしたことが確認できるからです。

## テストの実行 (Running Tests)

### 前提条件 (Prerequisites)

- Dockerがインストールされ、実行されていること。
- `make`がインストールされていること。

### コマンド (Command)

```bash
make test-integration
```

## メンテナンス (Maintenance)

### Dockerfile

`Dockerfile`は`node:22-slim`イメージをビルドし、サポートされているすべてのエージェントCLIをインストールします。

- 可能な限りネイティブインストーラーを使用します（Claude Code、Cursor）。
- 実際の開発者環境をシミュレートするために、非ルートの`testuser`を作成します。
- Electronベースのエージェント（Cursorなど）に必要なすべてのシステムライブラリを含みます。

### テストの追加 (Adding New Tests)

新しいテストケースを追加するには：

1.  `verify.sh`を開きます。
2.  Phase 1に新しい`run_test`呼び出しを追加するか、Phase 3に新しい`run_agent_test`呼び出しを追加します。
3.  `expected_exit_code`が、検証したい動作と一致していることを確認します。

## 自動メンテナンス (Automated Maintenance)

「テスト-分析-修正」ループを自動化するために、**[統合テスト修正ツール（Integration Test Fixer）](../.claude/skills/integration-test-fixer/SKILL.md)**エージェントスキルを使用します。このスキルは、一般的な失敗パターン（依存関係の欠如やパスの問題など）を特定し、修正を提案できます。
