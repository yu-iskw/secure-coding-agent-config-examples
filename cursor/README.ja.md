# Cursor セキュリティガイド

Cursor 2.0 では、macOS 上でエージェント・サンドボックス（Agent Sandboxing）が導入されました。これは、Apple のカーネルレベルの Seatbelt プリミティブを利用して、AI エージェントが実行するターミナルコマンドを安全に隔離するものです。本ガイドでは、AI エージェントによる不正な実行攻撃からマシンを保護するために、これらの設定を適切に構成する方法について説明します。

## 脅威

悪意のある VS Code または Cursor の拡張機能が、危険な権限を持つ AI エージェントを密かに呼び出そうとする可能性があります（Aqua Trivy 拡張機能における `--yolo` エクスプロイトを発見した Socket.dev のレポートなどがその例です）。チェックされていない AI エージェントは、ファイルシステムを探索したり、システムファイルを変更したり、インターネット経由で認証情報を外部に流出（エクスフィルトレーション）させたりする可能性があります。

## 解決策

エージェント・サンドボックスを有効にすると、Cursor は、良性なコマンドごとに手動の承認を求めることなく、エージェントのアクションを自動的に制限します。

### エージェント・サンドボックスの構成方法

Cursor エージェントを安全に構成するには、以下の手順に従ってください：

1. **Cursor Settings** を開きます。
2. **Agent** タブ（Features の下） -> **Auto-Run** に移動します。
3. **Auto-Run Mode** が **Run in Sandbox** に設定されていることを確認します。
   - _動作の内容:_ 強力な OS レベルのサンドボックス（macOS Seatbelt または Linux Landlock）を強制します。エージェントは、開いているワークスペースと `/tmp` に対してのみ読み書きが可能です。他のディレクトリへの書き込みや、`cursorignore` されたファイルへのアクセスはできません。
4. **Auto-run network access** トグルを **sandbox.json Only** に設定します。
   - _動作の内容:_ このモードに設定すると、エージェントは `sandbox.json` ファイルで明示的に許可されたドメインを除き、ネットワークにアクセスできなくなります。これにより、攻撃者が制御するサーバーへのデータ流出を防ぐことができます。
5. **External-File Protection** を **On** に設定します。
   - _動作の内容:_ エージェントがワークスペース外のファイルを自動的に作成したり変更したりするのを防ぎます。

### ネットワークアクセスの構成 (`sandbox.json`)

Cursor 2.5 では、`sandbox.json` を介したきめ細かなネットワークアクセス制御が導入されました。提供されている [`sandbox.json`](sandbox.json) を、プロジェクトの `.cursor/` ディレクトリ (`.cursor/sandbox.json`) またはグローバルな `~/.cursor/sandbox.json` に配置してください。

主なセキュリティ設定は以下の通りです：

- **`"networkPolicy": { "default": "deny" }`**: 許可リストのパターンに一致しない限り、サンドボックスからのすべてのアウトバウンドネットワークトラフィックを明示的に拒否します。
- **`"allow": [...]`**: パッケージレジストリ（npm、PyPI、Go、Crates など）や GitHub など、必要なドメインの厳格な許可リストです。これらのドメインのみがアクセス可能です。
- **`"deny": [...]`**: 社内サービスや特定の IP へのアクセスを明示的にブロックして、サーバーサイドリクエストフォージェリ（SSRF）攻撃や内部偵察を防ぐことができます。

### 粒度の細かいルールベースの制限 (`.cursor/rules`)

ネットワーク制限によって未知のドメインへのトラフィックをブロックできますが、`github.com` のような必要なドメインへのトラフィックは依然として許可される可能性があります。攻撃者はこれを利用して、エージェントに認証済みの `gh` CLI を使用させ、公開リポジトリや Gist にデータを公開させる可能性があります。

これを防ぐために、AI エージェントに対して明示的なコンテキストルールを提供できます。

`rules/security-cli-constraints.mdc` ファイルをプロジェクトの `.cursor/rules/` ディレクトリにコピーしてください。このルールは、エージェントが `gh repo create` や `gh gist create` といったコマンドを実行することを明示的に禁止し、安全なクエリ操作のみに CLI を使用するようにします。

### 粒度の細かいセキュリティフック (`.cursor/hooks.json`)

Cursor フック（Hooks）を使用すると、カスタムスクリプトを使用してエージェントループを監視、制御、拡張できます。これらはエージェントループの定義されたステージの前後に実行され、動作を監視、ブロック、または変更できます。

ルール（`.cursor/rules`）は AI の動作をガイドしますが、ハードな強制メカニズムではありません。AI は依然としてハルシネーション（幻覚）を起こしたり、ルールを無視するように仕向けられたりする可能性があります。**フックは強力な強制（Hard Enforcement）を提供します。**

これに対処するため、**`beforeShellExecution`** フックを使用して、シェルによって実行される前にコマンドをインターセプトします。

#### `gh-safeguard.sh` フックの使用方法

1. `hooks/gh-safeguard.sh` スクリプトをプロジェクトの `.cursor/hooks/` ディレクトリにコピーします。
2. スクリプトを実行可能にします：`chmod +x .cursor/hooks/gh-safeguard.sh`
3. `hooks.json` ファイルをプロジェクトの `.cursor/` ディレクトリ (`.cursor/hooks.json`) にコピーします。これにより、`gh` で始まるシェル実行の前にスクリプトが実行されるように登録されます。

このスクリプトは、LLM が実行しようとしているコマンドを解析し、`gh repo create` や `gh gist create` のような既知のデータ流出コマンドが含まれている場合は、厳格にブロックします（`{"permission": "deny"}` を返すことで）。一方で、安全なクエリコマンドの実行は許可します。

**リポジトリの固定（Repository Pinning）**: このフックは、`-R` および `--repo` フラグ（および `repos/owner/repo` のような明示的な API パス）もブロックします。これにより、`gh` CLI は強制的にローカルワークスペースのコンテキストを使用することになり、エージェントを現在のリポジトリに実質的に「固定」し、外部の攻撃者が制御するリポジトリにデータをプッシュすることを防ぎます。

## 検証

Cursor が正しく構成され、`gh-safeguard.sh` フックがアクティブであることを確認するには、リポジトリの統合テストスイートを使用できます：

```bash
make test-integration
```

これにより、自動化されたプロンプトが実行され、`agent -p 'gh repo create'` のようなコマンドが正常にブロックされることが確認されます。

### ストリクトモード（Strict Mode）の代替案

サンドボックス機能を使用しない場合や、まだサンドボックスを完全にサポートしていないモデルを使用している場合は、厳格な手動承認にフォールバックするように Cursor を構成できます：

- Cursor Agent の設定で、**Ask Every Time** を選択します。
  - _動作の内容:_ Cursor は一時停止し、ターミナルで _いかなる_ コマンドを実行する前にも、明示的な手動承認を求めます。
